# 프로젝트 규약

## 1. UI 컴포넌트 규칙
모든 UI 컴포넌트와 페이지에는 `data-ui-id` 속성을 부여한다.

### 네이밍 규칙
형식: `{타입}-{영역}-{용도}`

- 컴포넌트 예시:
  - `btn-login-submit` (로그인 영역의 제출 버튼)
  - `card-product-list` (상품 영역의 리스트 카드)
  - `input-signup-email` (회원가입 영역의 이메일 입력)
- 페이지 예시:
  - `page-home`
  - `page-product-detail`
  - `page-admin-dashboard`

### 관리자 hover 툴팁
- 현재 로그인한 사용자의 ID가 **DB에 등록된 관리자 ID**와 일치할 때만,
  컴포넌트에 마우스를 올리면 CSS 툴팁으로 `data-ui-id` 값을 말풍선으로 표시한다.
- 관리자 판별은 서버에서 검증된 세션/유저 정보를 기준으로 한다. (클라이언트 단독 판별 금지)
- 구현은 CSS `:hover` + `::after` 기반 툴팁을 사용한다. (외부 라이브러리 X)

## 2. 데이터그리드 / 리스트
- 데이터그리드, 테이블, 리스트 등 다건 데이터를 표시하는 컴포넌트에는 **페이징 처리를 의무화**한다.
- 기본 페이지 크기는 20건으로 하되, 화면 특성에 맞게 조정한다.
- 페이지 크기는 사용자가 조절할 수 있도록 **콤보박스(select)를 제공**한다.
  - 기본 선택지: `10 / 20 / 50 / 100`
  - 콤보박스 변경 시 1페이지로 이동하고, 새 페이지 크기로 데이터를 다시 조회한다.
  - 콤보박스에도 `data-ui-id` 부여 (예: `select-userlist-pagesize`)
- 페이지네이션 방식(번호 페이징 / 무한 스크롤 / 더보기 버튼)은 화면 용도에 맞게 제안하고 근거를 응답에 적는다.
- 서버 데이터는 **서버 사이드 페이징**을 기본으로 한다. (전체 조회 후 클라이언트에서 자르는 방식 금지)
- 정렬·필터링도 페이징과 함께 서버 사이드로 처리한다.

## 3. 화면(페이지) 관리 + 네비게이션
- 모든 페이지에는 `page-*` ID를 부여한다.
- 새 페이지를 만들면, 페이지 목록(라우트 맵)을 한 곳에서 관리하는 파일을 만들거나 갱신한다.
  - 예: `src/config/routes.ts` 또는 `src/config/pages.ts`
  - 각 항목에 `{ id, path, title, group }` 정도를 포함한다.
- 이 라우트 맵을 기반으로 네비게이션 컴포넌트를 구성한다.
  - 페이지를 추가할 때 라우트 맵만 갱신하면 네비게이션이 자동 반영되도록 한다.

## 4. SQL 파일 관리
- 모든 SQL은 **실행 순서대로** 별도 파일로 저장한다.
- 파일명 규칙: `{3자리 일련번호}_{동사}_{대상}.sql`
  - 예: `001_create_users.sql`, `002_add_index_orders.sql`, `003_alter_products_add_status.sql`
- 저장 위치:
  - Supabase 프로젝트: `supabase/migrations/`
  - 일반 프로젝트: `sql/` 또는 `db/migrations/`
- 모든 SQL 파일 상단에 주석으로 **실행하는 내용**을 한국어로 작성한다.
```sql
  -- users 테이블 생성: 회원가입 시 기본 정보 저장
  -- 작성일: YYYY-MM-DD
  CREATE TABLE users (...);
```
